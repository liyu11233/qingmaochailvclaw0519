# 青猫差旅航班比价第一版全栈技术路线图

日期：2026-05-18

## 1. 技术路线结论

第一版建议走本地优先架构。

核心形态：

- 用户本人电脑上运行管理采集端。
- 浏览器自动化使用已登录状态，不保存账号密码。
- 采集优先后台静默运行，不让浏览器窗口长期置顶影响用户其他任务。
- 主路径不使用 OCR，优先通过网页 DOM、页面元素、网络返回数据读取结构化信息。
- 数据和截图保存在本地项目目录。
- 每次手动采集生成一个独立批次。
- 当前批次可导出 Excel。
- 当前批次可导出离线销售演示包。

不要第一版就做云端 SaaS。原因是三家平台登录态、验证码、企业风控都在本机浏览器里更容易处理。

## 2. 推荐技术栈

### 2.1 前端

推荐：

- React
- TypeScript
- Vite
- Tailwind CSS 或等价轻量样式方案

理由：

- 管理端和离线演示包都需要网页体验。
- Vite 适合生成静态离线包。
- React 足够快，后续做组件复用方便。
- TypeScript 能减少字段错配，适合这种多平台采集项目。

前端分两套页面：

- 管理采集端：本机使用。
- 离线销售演示包：销售外出演示使用。

### 2.2 后端

推荐：

- Node.js
- TypeScript
- Fastify 或 Express

理由：

- 浏览器自动化、文件生成、Excel 导出都适合 Node 生态。
- 和前端同语言，第一版开发速度快。
- 本地服务足够，不需要复杂后端框架。

后端职责：

- 管理采集任务。
- 调用浏览器采集器。
- 保存结构化数据。
- 保存截图证据。
- 生成 Excel。
- 生成离线销售演示包。

### 2.3 浏览器采集

推荐：

- Playwright
- GStack Browser 作为可视化辅助和调试工具

第一版策略：

- 用户人工登录青猫差旅、携程商旅、阿里商旅。
- 系统复用已登录浏览器状态。
- 不做账号密码自动登录。
- 不保存账号密码。
- 优先使用 headless 后台采集。
- 如果平台阻止 headless，兜底使用 headed 浏览器非置顶或可最小化采集。
- 不用 OCR 作为主路径。
- 截图用于证据，不用于主要字段识别。

采集器按平台拆分：

- QingmaoAdapter
- CtripBusinessAdapter
- AlibabaBusinessAdapter

每个平台适配器只做三件事：

- 搜索指定航线和日期。
- 读取指定航班价格、退改、行李、是否可订。
- 截图保存证据。

### 2.4 本地数据

推荐：

- SQLite
- 本地文件目录保存截图和离线包

理由：

- 第一版只在用户电脑上跑。
- 数据量小。
- SQLite 足够稳定。
- 截图和离线包天然适合文件系统。

核心数据表：

- routes：固定航线。
- collection_batches：采集批次。
- flight_samples：随机抽中的航班。
- platform_quotes：三平台报价。
- evidence_files：截图证据。
- export_artifacts：Excel 和离线演示包记录。

### 2.5 Excel 导出

推荐：

- ExcelJS 或同类支持样式的 Excel 库。

理由：

- 第一版要求美观，不是简单 CSV。
- 需要多 sheet。
- 需要样式、列宽、冻结表头、重点价格突出。
- 需要证据索引。

Excel 结构：

- 汇总页。
- 证据索引页。
- 留痕页。

### 2.6 离线销售演示包

推荐形态：

- 一个文件夹。
- 内含静态网页。
- 内含结构化数据。
- 内含截图证据。
- 可压缩成 zip 发给销售。

建议目录形态：

- index.html：销售打开的入口。
- assets：样式和脚本资源。
- data：当前批次结构化数据。
- evidence：三平台截图。
- exports：Excel 文件。

销售只需要打开 index.html。

## 3. 系统数据流

### 3.1 管理采集流

1. 用户打开管理采集端。
2. 用户确认三平台已登录。
3. 用户点击“开始采集”。
4. 系统创建采集批次。
5. 系统读取 20 条固定航线。
6. 系统按航线打开青猫差旅。
7. 系统随机抽取同一航班。
8. 系统去携程商旅、阿里商旅查同一航班。
9. 系统保存报价和截图。
10. 系统生成对比结论。
11. 系统生成 Excel。
12. 系统生成离线销售演示包。

用户不需要手动选航班、截图、整理价格、生成结论、排版 Excel。

### 3.2 销售演示流

1. 用户把离线销售演示包给销售。
2. 销售复制到自己的电脑。
3. 销售打开 index.html。
4. 销售在客户面前点击演示页。
5. 页面展示随机同一航班三平台价格。
6. 客户质疑时，销售点击“查看证据”。
7. 会后销售发送 Excel。

## 4. 关键架构决策

### ADR 1：第一版本地优先，不做云端 SaaS

决策：

第一版运行在用户本人电脑上。

理由：

- 平台登录态在本机最容易处理。
- 不需要处理多用户权限。
- 不需要部署服务器。
- 不需要把敏感页面截图上传云端。

代价：

- 用户电脑需要运行管理端。
- 销售拿到的是离线包，不是实时在线页面。

结论：

第一版正确。等采集稳定后，再考虑云端。

### ADR 2：第一版人工登录，不做账号密码自动登录

决策：

系统只复用已登录浏览器状态。

理由：

- 商旅平台可能有验证码、扫码、企业 SSO、风控。
- 自动登录不稳定。
- 保存账号密码风险高。

代价：

- 用户需要先人工登录。

结论：

第一版必须这样做。否则项目会卡在登录层。

### ADR 5：第一版不使用 OCR 作为主采集方式

决策：

第一版主路径不使用 OCR。

理由：

- OCR 对页面缩放、字体、遮挡、滚动位置敏感。
- OCR 需要浏览器画面可见，容易影响用户其他工作。
- 网页 DOM、页面元素、网络返回数据更稳定。
- 截图证据可以保存，但不应该承担主要数据识别。

代价：

- 平台页面结构变化时，需要维护适配器。
- 某些字段如果只渲染在 canvas 或图片里，可能需要单独兜底。

结论：

正确。OCR 只作为未来极端兜底，不进入第一版主路径。

### ADR 6：采集优先后台静默运行

决策：

第一版优先 headless 后台采集。

理由：

- 用户只负责登录、点击采集、点击导出。
- 采集过程不应长期占用前台窗口。
- 后台运行更符合工具属性，不影响用户继续做其他任务。

代价：

- 部分平台可能检测 headless。
- 必须先用 1 条国内航线做可行性验证。

结论：

优先后台静默。若平台阻止，再切换 headed 非置顶或可最小化运行。

### ADR 3：销售端用离线包，不依赖管理端在线

决策：

销售外出演示使用离线包。

理由：

- 销售不会带用户电脑。
- 客户现场网络不可控。
- 销售不应该看到采集按钮。
- 离线包最稳。

代价：

- 数据不是实时刷新。
- 每次更新后需要重新发包给销售。

结论：

第一版适合离线包。

### ADR 4：结构化数据和截图证据必须绑定

决策：

每条平台报价都绑定截图证据。

理由：

- 销售材料必须可追溯。
- 客户质疑时能打开证据。
- 内部复盘有来源。

代价：

- 本地文件会变多。
- 离线包体积会变大。

结论：

必须做。没有证据，这个工具的可信度不够。

## 5. 模块路线图

### 5.1 数据模型模块

优先级：最高。

要解决：

- 固定航线。
- 采集批次。
- 随机航班。
- 三平台报价。
- 证据截图。
- 导出产物。

验收：

- 假数据能完整表达一个采集批次。

### 5.2 管理采集端

优先级：最高。

要解决：

- 最近一次采集状态。
- 开始采集。
- 采集进度。
- 成功/失败统计。
- Excel 导出。
- 离线包导出。

验收：

- 用户能看懂当前有没有可用数据。
- 用户能手动触发一次采集。

### 5.3 离线销售演示包

优先级：最高。

要解决：

- 离线打开。
- 美观展示。
- 随机展示同一航班。
- 三平台价格对比。
- 证据入口。
- 无采集按钮。

验收：

- 销售电脑断网也能打开。
- 客户现场能看懂价格差异。

### 5.4 Excel 导出

优先级：最高。

要解决：

- 汇总页。
- 证据索引页。
- 留痕页。
- 美观排版。

验收：

- Excel 可会后发送给客户。
- 内部可追溯。

### 5.5 浏览器采集适配器

优先级：第二阶段。

要解决：

- 青猫候选池。
- 携程商旅同航班查询。
- 阿里商旅同航班查询。
- 截图证据。
- 登录失效提示。
- 后台静默运行验证。
- 非 OCR 字段读取。

验收：

- 先跑通 1 条国内航线。
- 证明是否可以后台静默采集。
- 证明是否可以不用 OCR 获取主要字段。
- 再扩展 20 条航线。

## 6. 风险与处理

### 6.1 页面变化

风险：

平台页面结构可能变化。

处理：

- 平台适配器独立。
- 失败时保存截图。
- 不让单个平台失败拖垮整个批次。

### 6.2 登录失效

风险：

浏览器登录态过期。

处理：

- 管理端提示用户重新登录。
- 不自动输入账号密码。

### 6.3 价格口径不一致

风险：

三平台展示的价格可能含税费口径不同。

处理：

- 第一版按页面展示价格记录。
- 截图留证。
- 后续再做口径拆分。

### 6.4 离线包体积过大

风险：

截图太多，离线包变大。

处理：

- 第一版 20 条航线、3 平台截图，规模可控。
- 截图压缩。
- 证据页按需打开。

### 6.5 headless 被平台拦截

风险：

平台可能识别 headless 浏览器，导致搜索失败、验证码或空结果。

处理：

- 先用 1 条国内航线做验证。
- 如果 headless 不稳定，切换 headed 非置顶或可最小化运行。
- 保持不使用 OCR 的主路径。

## 7. 第二版方向

第一版稳定后，再考虑：

- 保存登录态后半自动采集。
- 定时提醒。
- 酒店价格对比。
- 打车价格对比。
- 普通 OTA 对比。
- 云端共享销售演示页。
- 批次历史趋势。
- 客户专属演示包。

不要提前做。

## 8. 技术路线最终判断

第一版最稳路线是：

- 本地 Node.js 后端。
- React 管理端。
- Playwright 后台静默优先的浏览器采集。
- SQLite 本地数据。
- 文件系统保存截图。
- ExcelJS 导出 Excel。
- Vite 静态构建离线销售演示包。

这个路线能最快跑通真实业务闭环。

确认后，第一步只做假数据骨架，不碰真实平台。
